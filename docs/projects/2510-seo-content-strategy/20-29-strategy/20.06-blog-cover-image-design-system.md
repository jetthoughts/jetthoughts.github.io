## 🎯 Design Intent In One Paragraph

Every jt_site blog cover image is a **dark IDE-style composition** built around one bold headline, a compact tag strip, and 2–3 metric or feature cards, rendered on an obsidian (`#10141a`) gradient background. Text is baked into the image at generation time — the Hugo post page renders an `<h1>` separately in the DOM above the cover, but the in-image title is the cover's primary affordance for social shares, Slack unfurls, and the blog list page. The aesthetic reads as confident technical authority: closer to GitHub Engineering blog, Vercel changelog, and Stripe Press than to stock photography or abstract 3D renders.

---

## 🧭 Why Obsidian Engine (not hand-drawn sketches)

Three independent expert reviews (UX researcher, editorial designer, frontend architect) converged on this direction for jt_site specifically:

1. **Audience fit**: CTOs, engineering managers, senior Rails developers reward information-forward covers. Their reference points are Hacker News and Lobsters, not Kinfolk. A metaphorical charcoal sketch next to "Rails 8.1 Active Job Continuations" forces the reader to decode before learning — technical readers penalize that cognitive tax.
2. **Sustainability at scale**: Metaphor-first illustration requires a human to invent a fresh physical/architectural metaphor for every post and iterate prompts until the generator produces something non-cliché. At 4–8 posts per month this is not scalable. A templated HTML → headless-browser-render pipeline produces a cover in seconds with deterministic typography and zero metaphor-search overhead.
3. **Shipped reality**: Two production covers (Ruby on Rails Performance Optimization 2026, Rails 8.1 Active Job Continuations) were already built in this style by parallel agent work before this doc was rewritten. The system was working; the spec had to catch up.

## 🎨 The Design System

### 1. Palette (locked — do not drift)

| Token | Hex | Usage |
|---|---|---|
| **Surface / background** | `#10141a` | Primary canvas — the deep obsidian base |
| **Surface container low** | `#181c22` | Large content blocks, sidebars |
| **Surface container** | `#1c2026` | Article cards, code blocks within the cover |
| **Surface container highest** | `#31353c` | Overlays, active states |
| **Primary / Ruby Red** | `#DC143C` → `#FF4757` gradient | Brand accent, highlighted headline words, ruby-tag pills |
| **Secondary / Neon Purple** | `#a855f7` / `#6f00be` | Gradient companions, interactive highlight hints (use sparingly) |
| **Tertiary / Cyan** | `#7dd0fe` | Accent lines, tech details (code block left borders) |
| **Headline text** | `#F8FAFC` | Never pure white — `#FFFFFF` reads harsh against obsidian |
| **Subtitle text** | `#CBD5E1` | Clearly readable against the dark gradient |
| **Metadata / label text** | `#94A3B8` | Tracked caps labels on metric cards, metadata lines |
| **Background code** | `rgba(148, 163, 184, 0.22)` | Faint JetBrains Mono snippets in negative space (NOT 0.15 — that was invisible; NOT 0.32 — that competes with title) |

### 2. Typography

| Role | Font | Size | Weight | Letter-spacing |
|---|---|---|---|---|
| **Display headline** (in-image title) | Space Grotesk | 56–64 px | 900 | `-0.02em` |
| **Highlighted phrase** (1–2 words max, not the whole headline) | Space Grotesk | 56–64 px | 900 | `-0.02em`, filled with `linear-gradient(135deg, #DC143C 0%, #FF4757 100%)`, `-webkit-background-clip: text` |
| **Subtitle** (in-image) | Inter | 22 px | 400 | default |
| **Tag pills** (in-image) | Inter | 15 px | 600 | `+0.5px` |
| **Metric values** (big numbers) | Space Grotesk or Inter | 42–48 px | 900 | `-0.01em` |
| **Metric labels** (tracked caps) | Inter | 13–14 px | 700, UPPERCASE | `+1.2px` |
| **Date** (bottom-right) | Inter | 18 px | 600 | default |
| **Background code snippets** | JetBrains Mono | 14–15 px | 400 | default |
| **Blog name** ("JETTHOUGHTS") | Inter | 24 px | 600, UPPERCASE | `+2px` |

Use Space Grotesk for display weight and aggressive character; Inter for everything else. JetBrains Mono only for code and numeric callouts. **Do not mix three display fonts**.

### 3. Composition (1200×630 OG standard)

| Rule | Value | Why |
|---|---|---|
| **Canvas size** | 1200 × 630 px | Open Graph / Twitter card standard |
| **Background** | `linear-gradient(135deg, #0F172A 0%, #1E293B 50%, #0F172A 100%)` with optional radial accent blobs (subtle, ≤0.18 opacity) | Deep but not flat; reads as dimensional IDE canvas |
| **Layout grammar** (top-to-bottom, left-aligned) | Blog name → Title → Subtitle → Tag pills → Metric cards → (Date bottom-right) | This order is locked — it is the editorial rhythm the system shipped with |
| **Padding** | 80 px all sides | Generous negative space; never let content touch edges |
| **Background code snippets** | `top: 60px; right: 80px;` — 4-5 lines, right-aligned, JetBrains Mono 15 px, `color: rgba(148,163,184,0.22)` | Tech authority signal without competing with the title |
| **Margin between sections** | 28–36 px vertical | Breathing room between title/subtitle, subtitle/pills, pills/metrics |
| **Ruby highlight scope** | 1–2 words ONLY, not the whole headline | Gradients at full-line scale read as Canva; keep them focal |
| **Tag count** | ≤ 3 pills (canonical is 2, max is 3) | "Tag soup" of 5+ pills dilutes the hierarchy |
| **Metric cards** | 2 or 3, never more | Boxed with `background: rgba(52, 211, 153, 0.08); border: 1px solid rgba(52, 211, 153, 0.25); border-radius: 12px; padding: 18px 24px` |

### 4. Content rules

The cover must answer, at thumbnail scale (300×157), two questions in under one second:

1. **What is this about?** — the headline plus optional highlighted key phrase
2. **Why should I care?** — the subtitle or the metric cards

Everything else is decoration. If the reader cannot answer both questions from a 300-wide preview, the cover has failed.

### 5. Absolute prohibitions

- **No stock photography** (server racks, keyboards, code-screen photos, developer hands-on-laptop)
- **No 3D renders** (abstract blobs, floating geometric shapes, metaverse-adjacent aesthetics) — see GitHub Engineering blog's weakest pre-2025 covers for the failure mode
- **No animated / video covers** — eye-tracking data shows motion captures fixation but reduces time-on-first-paragraph
- **No photography-style lighting effects** on the composition — directional rim lighting, drop shadows on UI elements, simulated ambient occlusion all fight the flat dark-IDE aesthetic
- **No pure white `#FFFFFF`** anywhere — always `#F8FAFC` or softer for text on dark
- **No more than ONE gradient per cover** (the headline highlight). Radial background accents are allowed but must stay ≤0.18 opacity
- **No watermarks, vendor logos, or "BLOG" banners**
- **No gradient overlays printed on top of a base image** to make text legible — the whole composition is text-on-dark from the start; this pattern would be for a photo cover and is not applicable here

## 🧱 The HTML Template

The canonical HTML template lives at [`.stitch/designs/rails-8-1-active-job-continuations-cover.html`](../../../../.stitch/designs/rails-8-1-active-job-continuations-cover.html) and is the reference implementation for every new cover. It uses inline `<style>` with Google Fonts CDN imports for Inter, Space Grotesk, and JetBrains Mono. New covers are produced by:

1. Copy the template
2. Swap the content blocks: blog name, headline, highlighted phrase, subtitle, tag pills, metric cards, background code snippets, date
3. Render at exact 1200 × 630 via `chrome-devtools` MCP at 2× effective resolution (set `document.documentElement.style.zoom = '2'`, resize viewport to 2400 × 1260, take PNG screenshot → 2× supersampled source)
4. Downsample via ImageMagick Lanczos to 960 × 504 at JPEG quality 95 with `-sampling-factor 4:4:4 -strip` (disables chroma subsampling — critical for the red ruby gradient on headline text)
5. Save to `content/blog/<slug>/cover.jpg`
6. Hugo's image pipeline takes it from there; the blog template renders it as a page resource via `Page.Resources.GetMatch "cover.*"`

This pipeline was validated in commit `641c97d55` which fixed a quality regression in the Rails 8.1 Active Job Continuations cover.

## 🧭 Blog Post Layout Integration

The cover image is rendered on every post page by the theme's `single.html` template via a cover hero block. The block is strictly additive and guarded with `{{ with .Resources.GetMatch "cover.*" }}` so posts without a local cover file render with no change.

Integration details as of 2026-04-09:

- **Position**: between the post `<header>` (title + tags + date) and the `.fl-rich-text` body. *May move above the header in a future revision per UX research recommendation.*
- **Max width**: `43rem` (matches the text column). Deliberately NOT full-bleed to 960 px or beyond — the text column alignment is a credibility signal for technical readers. Full-bleed hero images break that alignment and read as decoration-over-substance.
- **Responsive srcset**: three WebP variants (640 w, 960 w, 1920 w) let the browser pick an appropriate size and downsample using its own high-quality filter. This is how we bypass Hugo's `resampleFilter = "box"` setting without touching `hugo.toml`.
- **Box shadow**: `box-shadow: 0 1px 6px rgba(0,0,0,0.10)` so dark obsidian covers have a soft edge against the light page background (and so any future light-bg cover doesn't disappear into the page).
- **Caption** (optional): renders from a `cover_image_caption` frontmatter field inside a `<figcaption>` under the image. Used for attribution or a short metaphor-anchor line.

## 🛠 Cover Generation Requirements (for generator tooling)

Any tool that produces new covers — whether Stitch MCP, `bin/generate-cover-image`, or hand-authored HTML — must satisfy these requirements. Generators that cannot meet them are not used.

### Required

- **Dimensions**: exactly 1200 × 630 pixels at generation time
- **Final output**: JPEG at 960 × 504, quality 95, `-sampling-factor 4:4:4 -strip`, saved as `cover.jpg` in the target post directory. (The 2× render + Lanczos downsample pipeline is mandatory for text-bearing covers because it compounds with Hugo's pipeline to produce a readable result.)
- **Palette**: the tokens in Section 1 above. Do not invent new hex values. Do not replace the ruby accent with another color without updating this doc first.
- **Typography**: Space Grotesk + Inter + JetBrains Mono. All three must be loaded via Google Fonts or bundled locally. No system-font fallbacks accepted for the title.
- **Layout grammar**: blog-name → title → subtitle → tags → metrics → date, in that order. Any generator that can't produce this layout is not compliant.
- **Text baked in**: the cover IS the text composition. Generators that refuse to produce in-image text (or append "no text allowed" to the prompt) are fighting the system and must be rejected or rewritten.
- **Readable at 300 × 157**: the thumbnail scale is the acceptance test. If the title and at least one metric are not legible at that scale, the cover fails.

### Prohibited (generator-level)

- Stock photo APIs (Unsplash, Pexels) — the palette discipline cannot be enforced against external photography
- Generative image models that can't be steered to produce typography reliably (Imagen, Stable Diffusion, DALL-E 3 at the time of writing) — they are for illustration, not typography-first covers. If we need a background element (a subtle gem render, a vector shape), use a static asset library, not a generative model
- Any generator that doesn't expose the HTML intermediate — we need the HTML to re-render via headless browser for sharpness

### Recommended pipeline

1. Edit the canonical HTML template with the post's content
2. Open in chrome-devtools MCP via `file://` URL
3. Resize viewport to 2400 × 1260 and set `document.documentElement.style.zoom = '2'`
4. Wait for `document.fonts.ready` + short reflow settle (~200 ms)
5. `take_screenshot` as PNG (lossless) at 2× effective resolution
6. `magick src.png -filter Lanczos -resize 960x504 -quality 95 -sampling-factor 4:4:4 -strip cover.jpg`
7. Copy to post directory, rebuild Hugo, verify `og:image` resolves to the processed WebP (not the site-default fallback)

## 📚 Reference Publications

**Copy their discipline from:**

- **Stripe Press** — one color + one typeface + one hero form; ruthless minimalism on editorial book covers
- **GitHub Engineering blog** — dark backgrounds, single accent gradient, small category kicker, muted slate body text
- **Vercel changelog** — flat color field with wordmark, shockingly simple and instantly recognizable
- **Railway's blog** — dark gradient + single object render, almost Obsidian Engine but with more purple
- **Figma's blog** — generous type, one illustration, impossible to mistake for someone else's blog

**Patterns to deliberately resist:**

- **The Pragmatic Engineer** — ugly but works on brand equity jt_site doesn't have yet; don't copy Gergely's visual minimalism until jt_site has his audience
- **High Scalability** — stock server rack photos; the exact failure mode this system solves
- **HashiCorp, Vercel pre-2024, Cloudflare** — full-bleed cover images that break text column alignment; reads as decoration-over-substance on technical writing
- **HackerNoon / older dev.to** — author avatar + bio block interrupting the flow between cover and body; belongs at post footer, not masthead
- **Smashing Magazine / CSS-Tricks (RIP)** — gradient overlay printed on top of cover to make text legible. Our cover and title are separate DOM elements; the cover doesn't need legibility overlays

## 🔬 Known Limitations and Open Questions

1. **Hugo's `resampleFilter = "box"`** in `config/_default/hugo.toml:41` is the worst possible downsample filter for images with text. The srcset fix in `single.html` (three WebP widths — 640/960/1920) lets the browser do a better downsample and bypasses this for the cover specifically. Switching the site-wide setting to `Lanczos` or `CatmullRom` would help ~7500 processed images repo-wide, but would trigger a full image re-render, invalidate every CDN hash, and conflict with parallel agent commits. Deferred until a dedicated project slot; the cover-specific fix is sufficient for now.

2. **`bin/generate-cover-image` is currently misaligned with this spec**. The script today wraps the gemini CLI image model with a minimalist-hand-drawn prompt — leftover from the superseded direction. It should be rewritten to render the canonical HTML template via chrome-devtools MCP instead of calling Gemini at all. Tracked as a separate task; the script still works but produces the wrong style.

3. **Design system contradiction (resolved 2026-04-09)**: This file previously described a minimalist hand-drawn charcoal-on-cream editorial illustration direction. That direction was never shipped — every production cover used the Obsidian Engine instead. The contradiction was noted during a three-agent UX/visual/frontend review and resolved by enshrining Obsidian Engine as canonical. The minimalist research is preserved at the bottom of this file for historical context and may still inform future non-technical content (e.g., "soft" posts on team culture or hiring).

4. **Verification gap**: Multiple cover variants exist across shipped posts because parallel agents iterated on the system independently. The `.stitch/DESIGN.md` and `.stitch/design-system.md` files in scratch overlap with this doc. A consolidation pass is needed to produce a single canonical source. For now, when the files disagree, **this doc wins**.

## 🧭 How to Generate a New Cover (Checklist)

Run this sequence for every new post that needs a cover:

1. **Distill** the post into a 4–6 word headline and a 10–16 word subtitle. The subtitle must answer "why should the reader care?"
2. **Pick the highlighted phrase**: 1–2 words from the headline that will get the ruby-to-coral gradient. Not the whole headline.
3. **Pick 2 or 3 tags**: canonical tags only, no marketing fluff. Max 3 pills.
4. **Pick 2 or 3 metric cards**: real numbers from the post if possible (benchmarks, version numbers, counts). If the post has no numbers, use concept triples (e.g., "Resume / Retry / Skip" for a feature taxonomy).
5. **Pick 4–5 background code lines**: short, evocative snippets that hint at the topic. Real code from the post, not decorative.
6. **Edit the canonical HTML template** with these values.
7. **Render via chrome-devtools** at 2× / 2400 × 1260, take PNG.
8. **Downsample via magick Lanczos** to 960 × 504 JPEG q95 with 4:4:4.
9. **Save to** `content/blog/<slug>/cover.jpg`
10. **Rebuild Hugo**, verify `og:image` resolves to the processed WebP (not the site default).
11. **Eyeball test**: zoom out until the cover is 300 wide. Title readable? At least one metric readable? If yes, ship. If no, iterate.

## 📜 Historical Note: The Minimalist Research Brief (Superseded)

This section preserves the original research direction that was never shipped. Retained for context and in case jt_site ever spins up a separate content stream (e.g., team culture, hiring, non-technical posts) where a softer editorial illustration style would fit better than the Obsidian Engine. The prohibitions below are NOT in effect for current cover generation — see the Obsidian Engine spec above for the active rules.

### Original intent (archived)

Every jt_site cover image is a minimalist hand-drawn editorial sketch rendered in charcoal on cream paper, built around a single visual metaphor for the post's concept. No text, no logos, no UI chrome, no tech clichés. The aesthetic reads as confident editorial illustration, closer to a New Yorker spot illustration than a stock tech photo.

### Research basis (2025–2026)

- Hand-drawn sketches and line art trended heavily in 2025 as a counter-reaction to AI-generated digital output
- Minimalist single-focal-point designs outperform cluttered designs in engagement studies
- Custom thumbnails can lift CTR by up to 30 % ([LinkedIn study](https://www.linkedin.com/pulse/why-your-thumbnails-matter-look-data-behind-user-meer-shah--lpjjf))
- High-contrast designs perform ~20 % better than low-contrast ([Fastercapital](https://fastercapital.com/content/Reddit-post-thumbnail--The-Psychology-Behind-Effective-Reddit-Post-Thumbnails-for-Business-Success.html))
- Curved shapes enhance click-through rates ([Journal of Consumer Research](https://academic.oup.com/jcr/article/51/3/552/7470747))

### Original palette (archived)

- Background: `#FFF9F0` (cream) or `#F9F9F9` (near-white)
- Line art: `#333333` (charcoal)
- Stroke width: consistent 2–3 pt
- Paper grain texture for depth
- Hand-drawn imperfections encouraged
- 60–70 % negative space

### Why it didn't ship

- **Metaphor rule was self-defeating**: every post needed a human to invent a fresh metaphor and iterate prompts until the generator stopped producing tech clichés (code editors, gears, lightbulbs). Not sustainable at 4–8 posts/month.
- **Cream backgrounds disappeared into the page body**: the template's light background made near-white covers look like floating orphans with no visual container.
- **Audience mismatch**: technical readers responded better to dense information-forward covers than to whimsical metaphors.
- **Generator unavailability**: `bin/generate-cover-image` wrapped the gemini CLI image model, but the CLI's default OAuth endpoint (`cloudcode-pa.googleapis.com`) does not expose image generation models at all. Image models only live on `generativelanguage.googleapis.com` which requires a separate API key. This was documented in commit `2c19bb92d`. The script still exists but has never produced a working cover.
- **Parallel agents (Qwen-Coder) were already shipping Obsidian Engine covers** via Stitch MCP during the same session this file was originally written. The spec was diverging from reality in real time.

### What's worth keeping from this direction

Nothing for mainstream blog posts. But if jt_site ever publishes a culture/hiring/manifesto post where "technical authority" would feel cold, the minimalist hand-drawn direction is a valid alternative worth revisiting. At that point, create a new document (20.07 or similar) that documents the alternative system in its own right rather than trying to merge the two.
