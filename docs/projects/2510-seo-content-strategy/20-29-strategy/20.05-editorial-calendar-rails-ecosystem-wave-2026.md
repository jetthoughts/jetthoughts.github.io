## 🎨 VOICE & STRUCTURE GUIDE (derived from published reference post)

### Structural pattern (copy verbatim from the Active Job Continuations post)

1. **Punchy opening hook** — 1–2 sentences, often a contrarian claim or a named pain point
2. **Second paragraph bridging to the primary keyword** — MUST contain the full primary keyword phrase (SEO requirement; appeared in paragraph 3 of my first draft and had to be reworked)
3. **"Here's what changed / here's what fixes it" bridge** — sets up the body
4. **Numbered H2 sections** with action-oriented titles (e.g., "The Problem Rails 8.1 Just Solved", not "Problem Overview")
5. **Before/after code blocks** wherever a migration is involved — old pain in first block, new fix in second block, explanation between
6. **"When NOT to use this" section** — mandatory honesty bump, no exceptions
7. **Migration path** — 2–3 concrete numbered steps
8. **"The real win" closer** — reframes why the feature matters beyond the surface benefit
9. **"Related reading on this blog" paragraph** — 2 internal links to existing posts, in prose, not a list
10. **"Further reading" section** — 5 external citations as a bullet list with descriptive anchor text

### Frontmatter checklist (every post)

```yaml
---
title: "…"                       # ≤45 chars (theme truncates at 45 before appending " | JetThoughts Blog")
description: "…"                 # 150–160 chars, contains primary keyword
date: YYYY-MM-DDTHH:MM:SS+07:00  # use +07:00 to match reference
draft: false
author: "JetThoughts"
slug: "…"                        # explicit, hyphenated, no dots — drives the URL
keywords: "…, …, …"              # comma-separated SEO keywords (distinct from tags)
tags: ["…", "…"]                 # 6–8 lowercase hyphenated tags
categories: ["Engineering"]
cover_image: cover.jpg           # filename of a page-resource inside post dir
metatags:
  image: cover.jpg               # same filename; theme uses this for og:image
cover_image_alt: "…"
canonical_url: https://jetthoughts.com/blog/<slug>/
---
```

### Content rules (zero-tolerance — do not ship without each of these)

- **NO fabricated statistics.** Every number must trace to a cited source. When I first drafted #1 I invented a "40–70% compute reduction" claim that had no source; it was caught during SEO review. Do not repeat this mistake. If a plausible number doesn't have a citation, either find a citation or replace the claim with qualitative reasoning.
- **NO generic AI phrases**: dive in, unlock, in today's fast-paced, game-changer, journey, transform, seamless, robust, leverage (as a verb), boost, supercharge, revolutionize, embark, delve.
- **Paragraphs ≤3 sentences.** Breaks up the reading flow and reads like a human.
- **Word count 1,200–1,800.** Longer than that is a PILLAR article and belongs in the AI rotation, not this wave.
- **4+ contextual internal links** to existing posts (in prose, not footer lists).
- **5+ external citations** in the "Further reading" footer, preferring official docs.
- **Primary keyword in first 100 words of body** (not just the H1).
- **At least one before/after code example.**
- **Mandatory "When NOT to use this" section.**

### Pre-commit validation checklist

```bash
# 1. Build passes
./bin/hugo-build

# 2. Single canonical URL (no stale slugged versions)
find _dest/public-dev/blog -maxdepth 1 -type d -name "*<key-term>*"

# 3. Start dev server
./bin/hugo-dev &

# 4. Chrome DevTools validation (use chrome-devtools MCP)
# - Page returns HTTP 200
# - Zero console errors
# - <title> is NOT truncated (no trailing "…")
# - og:title and twitter:title match <title>
# - Meta description is 150–160 chars
# - Primary keyword present in first 100 words
# - All internal links return 200 (fetch HEAD via page context)
# - Schema.org Article JSON-LD present and populated
# - No broken images INSIDE the article (theme chrome icons are pre-existing and OK)
# - No fabricated percentage claims (grep the body text)

# 5. Stop dev server and clean up
pkill -f "hugo server"
```
